# 每日学习文件编写规范

版本：1.0  
适用范围：本工作区内所有机器人学习项目。  
目标：让每天的学习材料可执行、可复现、可验收，并能持续转化为 CV、个人网页和面试内容。

后续生成每日学习文件时，以本规范为默认依据。若某个项目有特殊要求，应在对应项目 README 中明确说明偏离原因。

## 1. 每日学习产物

一个完整学习日通常包含：

```text
学习计划 Markdown
-> 引导式 Notebook
-> 可运行 Python 脚本
-> Figures / Results
-> 学习结果总结
```

不同阶段的要求：

| 阶段 | 必须产物 | 说明 |
|---|---|---|
| 开始学习前 | 计划 MD、Notebook 框架 | 明确任务、参考代码、实现顺序和验收标准 |
| Notebook 学习中 | 已填写 Notebook | 按单元格逐步实现、检查中间变量和图表 |
| Notebook 跑通后 | Python 脚本 | 将 notebook 整理成可重复运行的正式脚本 |
| 当天结束时 | 图、CSV、总结 | 记录真实参数、结果、问题和理解 |

不应在概念尚未理解时一次性生成全部复杂代码。默认顺序是先框架、再填写、再整理脚本。

## 2. 文件命名规则

### 2.1 学习计划 MD

```text
notes/dayNN_topic.md
```

示例：

```text
notes/day05_ekf_unicycle_model.md
```

`NN` 表示项目内学习日编号，使用两位数字。

### 2.2 Notebook

```text
notebooks/NN_topic.ipynb
```

Notebook 编号按项目技术任务顺序连续排列，不重复已有编号。

示例：

```text
notebooks/04_ekf_unicycle_model.ipynb
```

### 2.3 Python 脚本

```text
scripts/NN_topic.py
```

Python 脚本编号和对应 notebook 保持一致：

```text
notebooks/04_ekf_unicycle_model.ipynb
scripts/04_ekf_unicycle_model.py
```

### 2.4 结果文件

```text
figures/topic_trajectory.png
figures/topic_error.png
results/topic_rmse.csv
```

文件名应描述内容，不使用 `result1.png`、`test.csv` 等模糊名称。

## 3. 学习计划 MD 编写规则

每个计划 MD 至少包含以下部分。

### 3.1 标题和学习边界

必须写明：

- Day 编号和主题。
- 本日目标。
- 最终交付物。
- 今天明确不做什么。

“不做什么”用于控制学习范围，避免一天内同时加入过多新概念。

### 3.2 前置条件

必须明确：

- 需要完成哪个前一天任务。
- 需要理解哪些概念。
- 需要哪些依赖或环境。

### 3.3 参考代码位置

必须给出具体文件路径和需要查看的函数，而不是只写“看官方代码”。

正确示例：

```text
source/filterpy/filterpy/kalman/EKF.py
只查看 ExtendedKalmanFilter.update、HJacobian、Hx 的调用方式。
```

错误示例：

```text
学习 FilterPy 源码。
```

### 3.4 理论模型

必须写明：

- 状态向量及顺序。
- 控制输入或观测向量。
- 运动模型。
- 观测模型。
- 矩阵和数组形状。
- 噪声参数的含义。

状态顺序必须在 MD、Notebook、Python、图表和总结中完全一致。

### 3.5 函数拆分

每个函数写明：

- 函数名。
- 输入。
- 输出。
- 数组形状。
- 关键逻辑。
- 对应的数学意义。

### 3.6 推荐实现顺序

任务必须拆成可验证的小节点：

```text
先生成 ground truth
-> 检查传感器数据
-> 检查模型/Jacobian
-> 实现滤波循环
-> 计算指标
-> 绘图
-> 整理脚本
```

每个节点都要有检查方法，不能只在最后看一张图判断成败。

### 3.7 常见错误

至少列出：

- 矩阵维度错误。
- 状态顺序错误。
- 时间索引错位。
- 传感器缺失值处理错误。
- 结果文件被占用。
- 参数含义混淆。

### 3.8 验收标准

验收分三类：

```text
文件验收：脚本、图片、CSV 是否存在。
结果验收：指标是否支持预期结论。
理解验收：是否能回答核心问题。
```

## 4. Notebook 框架编写规则

### 4.1 单元格结构

Notebook 使用交替结构：

```text
Markdown：解释当前任务、输入输出、参考位置
Code：实现一个功能并检查结果
Markdown：解释下一个任务
Code：继续实现
```

每个 code cell 只承担一个清晰职责。

### 4.2 框架层级

新主题第一次生成 notebook 时：

- 导入、路径、参数可以直接给出。
- 已在前一天掌握的通用函数可以给出完整实现。
- 当天核心新知识使用 `TODO` 引导，不直接隐藏关键步骤。
- 每个 TODO 必须说明输入、输出和检查方式。

### 4.3 可复现性

Notebook 必须：

- 固定 `random_seed`。
- 明确项目根目录。
- 将输出写入 `figures/` 和 `results/`。
- 在完成后执行 `Restart Kernel and Run All`。
- 不依赖之前手工运行遗留的隐藏变量。

Notebook 中显示的旧输出不等于当前代码结果。代码修改后必须重新完整运行。

### 4.4 中间检查

关键数组必须打印形状：

```text
ground truth
measurements
states
F/H/P/Q/R
availability mask
```

先检查数据和维度，再绘图。

### 4.5 Notebook 完成标准

- 所有必要 cell 能从上到下运行。
- 没有未解释的异常。
- 没有仍需完成的核心 TODO。
- 图片和 CSV 成功保存。
- 最终输出与总结文件一致。

## 5. Python 脚本编写规则

Python 脚本应在 notebook 跑通后整理。

### 5.1 基本结构

```python
from pathlib import Path

PROJECT_ROOT = Path(__file__).resolve().parents[1]

def function_a():
    ...

def function_b():
    ...

def main():
    ...

if __name__ == "__main__":
    main()
```

### 5.2 编码要求

- 不依赖 notebook 隐藏状态。
- 参数集中放在 `main()` 或配置区。
- 函数有明确输入和返回值。
- 不保留无意义的单独变量显示语句。
- 输出目录自动创建。
- 保存图片后调用 `plt.close(fig)`。
- 终端打印关键指标和保存路径。
- 固定随机种子，保证可复现。

### 5.3 与 Notebook 一致

Python 脚本必须与已完成 notebook 保持一致：

- 相同状态顺序。
- 相同参数。
- 相同随机数调用顺序。
- 相同时间索引。
- 相同 RMSE 定义。
- 相同输出文件名。

如果结果不同，先检查 notebook 是否保存了旧输出，再检查随机数和索引逻辑。

## 6. 学习结果总结规则

总结可以放在计划 MD 末尾，也可以单独建立：

```text
notes/dayNN_topic_summary.md
```

复杂任务建议单独建立总结文件。

总结必须填写实际结果，不复制计划中的预期结果。

至少包含：

- 完成日期和投入时间。
- 实际参数。
- 数组和矩阵形状。
- 运行问题及解决方法。
- 图片分析。
- RMSE 或成功率。
- 参数影响。
- 当前简化假设和局限。
- 必答问题。
- 中文和英文面试表达。

数值必须带单位或百分号。例如：

```text
RMSE 降低百分比：54.96%
```

不能写成：

```text
RMSE 降低百分比：0.5496
```

## 7. 一致性检查清单

每天结束前逐项检查：

```text
[ ] MD、Notebook、Python 的状态顺序一致
[ ] 参数值一致
[ ] 函数名和输出文件名一致
[ ] random_seed 一致
[ ] 时间索引一致
[ ] 矩阵形状正确
[ ] Notebook 已 Restart Kernel and Run All
[ ] Python 脚本可从项目根目录运行
[ ] 图表和 CSV 已生成
[ ] CSV/图片不是旧运行结果
[ ] 总结中的数字来自当前 CSV
[ ] 面试表达与实际完成内容一致
```

## 8. 文件占用和运行问题

Windows 中，如果 CSV 正在 Excel 打开，或图片正在某些查看器中打开，脚本覆盖时可能出现：

```text
PermissionError: [Errno 13] Permission denied
```

处理方法：

1. 关闭 Excel、图片查看器或相关预览。
2. 重新运行 notebook 或 Python 脚本。
3. 检查文件修改时间是否更新。

不要把文件占用错误误判为算法错误。

## 9. 每日完成流程

稳定执行顺序：

```text
1. 阅读当天计划 MD。
2. 打开 notebook，按顺序完成 TODO。
3. 检查中间数组、矩阵和传感器数据。
4. Restart Kernel and Run All。
5. 确认图和 CSV 更新。
6. 将 notebook 整理成 Python 脚本。
7. 从项目根目录独立运行 Python 脚本。
8. 对比 notebook 和 Python 结果。
9. 填写学习结果总结。
10. 练习中英文项目表达。
```

## 10. CV 和面试约束

只有达到以下标准，才能将技能写入 CV：

- 有可运行代码。
- 有图表或量化指标。
- 能解释输入、输出和核心方法。
- 能说明为什么方法有效。
- 能说明当前局限。
- 能回答如何扩展到真实机器人系统。

不能只因为安装或调用了某个库，就在 CV 中声称掌握该方向。
